---\r\ncanonical_id: 01KQ0P44TNFQMPBHGY5R4G4A4F\r\ncluster: databases\r\nsummary: End-to-end guide for bringing up a fresh Wikantik instance, including how to import pages from a legacy JSPWiki deployment.\r\ntags:\r\n- deployment\r\n- postgresql\r\n- operations\r\ntype: article\r\n---\r\n\r\n# Fresh Deployment \u0026 Legacy Article Import\r\n\r\nThis guide walks you through standing up a new Wikantik instance against PostgreSQL and, optionally, importing an existing corpus from a legacy JSPWiki deployment. It reflects the current code on `main`: database migrations live in `bin/db/migrations/`, the WAR deploys as the ROOT context, and the React SPA is served from the same origin.\r\n\r\n## Overview\r\n\r\nA complete deployment touches five things:\r\n\r\n| Layer | What it holds | How it is provisioned |\r\n|-------|---------------|-----------------------|\r\n| PostgreSQL | Users, roles, groups, policy grants, chunk embeddings, API keys | `bin/db/install-fresh.sh` + `bin/db/migrate.sh` (idempotent) |\r\n| Tomcat 11 | Servlet container + JNDI DataSource | Downloaded automatically by `bin/deploy-local.sh` |\r\n| Page corpus | Markdown files under `docs/wikantik-pages/` | Version-controlled; no per-instance copy step |\r\n| Search indexes | Lucene (in `workDir`) + `content_chunk_embeddings` (Postgres) | Built on startup + async bootstrap |\r\n| Embedding backend | Ollama HTTP endpoint (optional) | External service; URL configured in properties |\r\n\r\n---\r\n\r\n## Prerequisites\r\n\r\n| Tool | Version | Check |\r\n|------|---------|-------|\r\n| Java (JDK) | 21+ | `java -version` |\r\n| Maven | 3.9+ | `mvn -version` |\r\n| Node.js + npm | 18+ | Required — the WAR build runs `npm install` + `vite build` automatically |\r\n| PostgreSQL | 15+ (with `pgvector`) | `psql --version`; pgvector is installed by `V004` |\r\n| `curl`, `tar` | — | Used by `deploy-local.sh` to fetch Tomcat |\r\n\r\nPostgreSQL must be reachable on `localhost:5432` (or override `PGHOST`/`PGPORT`).\r\n\r\n---\r\n\r\n## Fresh Deployment (Clean Instance)\r\n\r\n### 1. Bootstrap the database\r\n\r\n`install-fresh.sh` creates the database, the application role, and runs every migration in order. It is idempotent — re-running against an already-bootstrapped database is a no-op.\r\n\r\n```bash\r\nsudo -u postgres \\\r\n    DB_NAME\u003dwikantik \\\r\n    DB_APP_USER\u003djspwiki \\\r\n    DB_APP_PASSWORD\u003d\u0027ChangeMe123!\u0027 \\\r\n    bin/db/install-fresh.sh\r\n```\r\n\r\nThis creates:\r\n\r\n- Database `wikantik` with the `pgvector` extension installed\r\n- Application role `jspwiki` with `CONNECT` + `USAGE` on `public`\r\n- All tables from `bin/db/migrations/V001`–`V010` (users/roles/groups, policy grants, knowledge graph, content chunks + embeddings, API keys)\r\n- A `schema_migrations` table so `migrate.sh` knows what has been applied\r\n\r\nTo bootstrap a dedicated migration role (recommended for production so `ALTER` migrations don\u0027t run as `postgres`), also set `DB_MIGRATE_PASSWORD` and the script will call `create-migrate-user.sh` and transfer ownership.\r\n\r\n### 2. Build the WAR\r\n\r\n```bash\r\nmvn clean install -Dmaven.test.skip -T 1C\r\n```\r\n\r\nThis compiles all modules, builds the React frontend, and produces `wikantik-war/target/Wikantik.war`. Use a full `mvn clean install` (without `-Dmaven.test.skip`) before any production cutover to make sure the test suite is green.\r\n\r\n### 3. Deploy to Tomcat\r\n\r\n```bash\r\nbin/deploy-local.sh\r\n```\r\n\r\nThe script is self-assembling. On first run it will:\r\n\r\n1. Download Tomcat 11 into `tomcat/tomcat-11/` (gitignored)\r\n2. Download the PostgreSQL JDBC driver to `tomcat/tomcat-11/lib/`\r\n3. Copy the context template to `tomcat/tomcat-11/conf/Catalina/localhost/ROOT.xml` with a `YOUR_SECURE_PASSWORD_HERE` placeholder\r\n4. Copy the properties template to `tomcat/tomcat-11/lib/wikantik-custom.properties`, substituting `@@REPO_ROOT@@` for the repo path\r\n5. Copy the Log4j config to `tomcat/tomcat-11/lib/log4j2.xml`\r\n6. Deploy `Wikantik.war` as `webapps/ROOT.war`\r\n7. Run `migrate.sh` against the target database (no-op on fresh install, applies any new migrations on subsequent runs)\r\n8. Run `bin/db/seed-users.sql` — upserts the `admin` / `admin123` and `jakefear@gmail.com` / `passw0rd` dev accounts\r\n9. Start Tomcat\r\n\r\n### 4. Set the database password\r\n\r\nEdit the context file to replace the placeholder:\r\n\r\n```bash\r\n$EDITOR tomcat/tomcat-11/conf/Catalina/localhost/ROOT.xml\r\n# username\u003d\"wikantik\" → username\u003d\"jspwiki\"\r\n# password\u003d\"YOUR_SECURE_PASSWORD_HERE\" → password\u003d\"ChangeMe123!\" (or whatever you set)\r\n```\r\n\r\nRestart Tomcat so the DataSource picks up the new password:\r\n\r\n```bash\r\ntomcat/tomcat-11/bin/shutdown.sh\r\ntomcat/tomcat-11/bin/startup.sh\r\n```\r\n\r\n### 5. Verify\r\n\r\n- Browse http://localhost:8080/ — the React SPA should load.\r\n- Log in as `admin` / `admin123` (change this immediately if the instance is anything more than local scratch).\r\n- Visit `/admin/content/stats` from the admin UI — confirm page count and index status.\r\n- Tail `tomcat/tomcat-11/logs/catalina.out` for any DataSource errors.\r\n\r\nFor automated testing, `test.properties` (gitignored) holds a dedicated `testbot` admin account — see `CLAUDE.md` for how to recreate it after a database reset.\r\n\r\n---\r\n\r\n## Importing Articles from a Legacy JSPWiki Deployment\r\n\r\nWikantik stores pages as `*.md` files under `docs/wikantik-pages/` (version-controlled), with optional `*.properties` sidecars for per-page metadata. A legacy JSPWiki corpus stores pages as `*.txt` files using classic wiki syntax. The import is a one-time conversion.\r\n\r\n### 1. Export the legacy corpus\r\n\r\nCopy the page directory and attachments from the old deployment:\r\n\r\n```bash\r\n# On the legacy host\r\ntar -czf legacy-pages.tgz \\\r\n    /var/lib/jspwiki/pages \\\r\n    /var/lib/jspwiki/attachments\r\n\r\nscp legacy-pages.tgz new-host:/tmp/\r\n```\r\n\r\nA JSPWiki page directory typically contains:\r\n\r\n- `PageName.txt` — the current revision, wiki syntax\r\n- `PageName.properties` — frontmatter-like metadata (author, timestamp, keywords)\r\n- `OLD/PageName/1.txt`, `2.txt`, ... — prior revisions (optional; Wikantik treats git history as the version store)\r\n- `attachments/PageName-att/...` — file attachments per page\r\n\r\n### 2. Stage the files\r\n\r\nUnpack into a staging directory — **not** directly into `docs/wikantik-pages/` until after conversion:\r\n\r\n```bash\r\nmkdir -p /tmp/wikantik-import\r\ntar -xzf /tmp/legacy-pages.tgz -C /tmp/wikantik-import --strip-components\u003d3\r\n```\r\n\r\nDrop the `OLD/` hierarchy if you do not want to preserve old revisions as separate pages — Wikantik treats git as the version of record.\r\n\r\n### 3. Convert wiki syntax → Markdown\r\n\r\nThe `scripts/wiki2markdown.py` converter is a faithful port of `WikiToMarkdownConverter.java`. It scans a directory for `.txt` files, scores each against a wiki-syntax heuristic, converts if the score is above threshold, renames straight to `.md` if the file is already Markdown, and deletes the source `.txt`.\r\n\r\n```bash\r\n# Dry-run first — prints every planned action with per-file warnings\r\npython3 scripts/wiki2markdown.py /tmp/wikantik-import --dry-run\r\n\r\n# Commit the conversion\r\npython3 scripts/wiki2markdown.py /tmp/wikantik-import\r\n```\r\n\r\nConversions performed:\r\n\r\n| Wiki | Markdown |\r\n|------|----------|\r\n| `! Heading`, `!! Heading`, `!!! Heading` | `# Heading`, `## Heading`, `### Heading` |\r\n| `__bold__` | `**bold**` |\r\n| `\u0027\u0027italic\u0027\u0027` | `*italic*` |\r\n| `{{inline code}}` | `` `inline code` `` |\r\n| `[PageName]` | `[PageName]()` (Flexmark plugin syntax — Wikantik resolves the empty URL on render) |\r\n| `[text\\|url]` | `[text](url)` |\r\n| `[{Plugin args}]` | `[{Plugin args}]()` |\r\n| `{{{ code block }}}` | triple-backtick fenced block |\r\n| `\\|\\| header \\|\\| cells \\|\\|` / `\\| row \\|` | GFM tables |\r\n\r\nAny unconverted constructs (`%%style`, old-style footnotes, etc.) are flagged as warnings next to the filename. Resolve these by hand or leave them — Flexmark will render them as literal text.\r\n\r\n### 4. Move into the page corpus\r\n\r\n```bash\r\n# Pages\r\ncp /tmp/wikantik-import/*.md       docs/wikantik-pages/\r\n# Keep the property sidecars — they carry categories, keywords, authors\r\ncp /tmp/wikantik-import/*.properties docs/wikantik-pages/ 2\u003e/dev/null || true\r\n```\r\n\r\nAttachments are not yet version-controlled in `docs/wikantik-pages/`. If the legacy corpus uses them:\r\n\r\n```bash\r\n# Attachments live under the wiki workDir, referenced by wikantik.basicAttachmentProvider.storageDir\r\nmkdir -p tomcat/tomcat-11/data/attachments\r\ncp -R /tmp/wikantik-import/attachments/* tomcat/tomcat-11/data/attachments/\r\n```\r\n\r\n### 5. Backfill frontmatter (optional)\r\n\r\nWikantik expects a YAML frontmatter block on each Markdown page. If a page lacks one, `wikantik.frontmatter.autoDefaults \u003d true` (set by the deploy template) will generate a minimal block — `title`, `type: article`, `tags: [uncategorized]`, empty `summary` — the first time the page is saved through the UI. For a bulk import you typically want real frontmatter. A minimal block looks like:\r\n\r\n```yaml\r\n---\r\ntype: article\r\ntags:\r\n- imported\r\nsummary: One-line summary for search and link previews.\r\n---\r\n```\r\n\r\nRun a small script over the staging directory to inject this block on every `.md` file that does not already have one — the import script at `scripts/wiki2markdown.py` does **not** do this for you.\r\n\r\n### 6. Commit and redeploy\r\n\r\n```bash\r\ngit add docs/wikantik-pages/\r\ngit commit -m \"import: legacy JSPWiki page corpus\"\r\nbin/deploy-local.sh\r\n```\r\n\r\n`deploy-local.sh` does not rebuild indexes automatically — Wikantik will pick up the new Markdown on first request, but Lucene and the chunk embeddings need to be rebuilt explicitly.\r\n\r\n### 7. Rebuild indexes\r\n\r\n**Lucene + chunks** — via admin UI at `/admin/index-status`, or via REST:\r\n\r\n```bash\r\nsource \u003c(grep -v \u0027^#\u0027 test.properties | sed \u0027s/^test.user.//\u0027 | sed \u0027s/\u003d/\u003d\"/\u0027 | sed \u0027s/$/\"/\u0027)\r\n\r\ncurl -u \"${login}:${password}\" -X POST \\\r\n     http://localhost:8080/admin/content/rebuild-indexes\r\n```\r\n\r\n**Embedding reindex** (hybrid search — only if `wikantik.search.hybrid.enabled\u003dtrue` and the Ollama endpoint is reachable):\r\n\r\n```bash\r\ncurl -u \"${login}:${password}\" -X POST \\\r\n     http://localhost:8080/admin/content/reindex-embeddings\r\n```\r\n\r\nThe embedding rebuild is async and streams progress into `GET /admin/content/index-status`. For a corpus of a few thousand pages against a CPU-only Ollama host, budget 30–90 minutes.\r\n\r\nOn a completely fresh instance the startup `BootstrapEmbeddingIndexer` will kick off the embedding run automatically when it detects an empty `content_chunk_embeddings` table — no admin action needed unless you want to force a rerun.\r\n\r\n---\r\n\r\n## Subsequent Deployments\r\n\r\nAfter the one-time setup, redeploying is a single command:\r\n\r\n```bash\r\nbin/deploy-local.sh\r\n```\r\n\r\nThe script is safe to re-run: it preserves the context file (and therefore the DB password), reapplies any new migrations, re-seeds users, and restarts Tomcat.
