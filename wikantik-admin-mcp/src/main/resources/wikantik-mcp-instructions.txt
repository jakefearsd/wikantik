You are connected to a Wikantik knowledge base — a semantic wiki that stores pages
as Markdown with YAML frontmatter for structured metadata.

== Page format ==

Every page may begin with a YAML frontmatter block fenced by --- lines, followed
by a Markdown body:

    ---
    type: report
    tags: [security, quarterly]
    date: 2026-01-15
    related: [SecurityPolicy, ThreatModel]
    status: draft
    summary: Q1 security review findings
    ---
    # Quarterly Security Report
    Body content in Markdown ...

Standard frontmatter fields (all optional):
  type        — page classification (report, note, reference, runbook, hub, ...)
  tags        — list of topic tags for categorisation
  date        — ISO date associated with the page
  related     — list of CamelCase page names this page relates to
  status      — lifecycle status (draft, active, archived)
  summary     — one-line description of the page
  verified_at — ISO timestamp recording the last human verification (set by mark_page_verified)
  verified_by — login name of the verifier
  confidence  — authoritative | provisional | stale (usually computed; can be pinned)
  audience    — humans | agents | [humans, agents]

You may add custom fields beyond these; they will be preserved and queryable.

== Metadata and web visibility ==

Frontmatter fields directly drive search engine, social sharing, feed, and news
outputs. The mapping is:

  Field     Web Output
  -----     ----------
  summary   <meta description>, OG/Twitter description, JSON-LD description,
            Atom <summary>
  tags      <meta keywords>, article:tag per tag, JSON-LD keywords,
            Atom <category>, News Sitemap <news:keywords>
  cluster   JSON-LD articleSection, isPartOf (non-hub), BreadcrumbList (non-hub)
  type      JSON-LD @type — hub=CollectionPage, else Article
  date      JSON-LD datePublished
  related   JSON-LD hasPart (hub) or relatedLink (non-hub)

Summary length: keep summary between 50 and 160 characters. Google truncates
meta descriptions at roughly 155 characters. Summaries under 50 characters
waste SERP real estate.

Tags and news eligibility: pages modified within the last 2 days that have
frontmatter tags appear in the Google News Sitemap. Pages without tags never
appear, regardless of recency.

Cluster/type interaction: hub pages (type=hub) emit CollectionPage JSON-LD with
hasPart listing their related pages. Non-hub clustered pages emit BreadcrumbList
JSON-LD (Home > Cluster > Page) and isPartOf referencing the collection.

== Page naming ==

Page names use CamelCase (e.g. SecurityPolicy, ProjectAlpha, MeetingNotes2026).
Names are case-sensitive and must not contain spaces.

== Frontmatter normalization on write ==

You can pass frontmatter to write_pages and update_page in either of two ways:

  1. As structured fields in the metadata argument (preferred). The server
     emits well-quoted YAML automatically — values like
     "Woodworking Joinery: Structural Mechanics" are quoted for you.

  2. Embedded in the content string as a YAML frontmatter block (---) at the
     top. The server parses it, merges it with any explicit metadata argument
     (explicit wins on conflict), and re-emits clean YAML on save. Either
     style works; the structured metadata form skips the parse step entirely.

If the embedded YAML is malformed, the tool returns a structured error with
the SnakeYAML message and best-effort line/column, plus a hint. Common case:

  title: Woodworking Joinery: Structural Mechanics

The unquoted colon makes YAML parse 'title' as a nested mapping start. Fix
either by quoting the value:

  title: "Woodworking Joinery: Structural Mechanics"

or by passing the field via the structured metadata argument:

  metadata: { "title": "Woodworking Joinery: Structural Mechanics" }

The same validation runs as a save-time filter on every write path
(REST, JSP editor, direct provider) — pages with malformed frontmatter are
rejected, not saved with empty metadata.

== System pages are read-only via MCP ==

The wiki ships with system pages — CSS theme pages (CSSRibbon, CSSPrettifyTheme*,
CSSBackgroundGradients, ...), menu fragments (LeftMenu, LeftMenuFooter), help pages
(EditPageHelp, SearchPageHelp, TextFormattingRules, ...), Main, and any pages
matched by the operator's wikantik.systemPages.extraPatterns regex. These pages
underpin the rendering surface of every wiki page; rewriting them via an agent
would let style or layout drift across the whole site without an admin in the
loop. Every write tool on this server refuses system-page targets:

  write_pages         → per-page error "system page — refusing to write via MCP"
  update_page         → {updated:false, error:"system page — refusing to edit via MCP"}
  delete_pages        → per-page error "system page — refusing to delete"
  rename_page         → top-level error on either oldName or newName
  mark_page_verified  → per-page error "system page — refusing to stamp via MCP"

If you genuinely need to change a system page, route the work through the admin
UI / direct database access — not the MCP surface. Read access (read_page,
get_backlinks, verify_pages, preview_structured_data, etc.) is unaffected.

== Link format ==

Use Markdown link syntax for internal links: [link text](PageName)
The target is a CamelCase page name, not a URL. Examples:
  [security policy](SecurityPolicy)
  [see the threat model](ThreatModel)

== Tool usage guide ==

Read tools

read_page         Read the raw Markdown body and YAML frontmatter of a page.
                  Returns {exists, pageName, content, contentHash, version,
                  lastModified}. Pass the contentHash back to update_page as
                  expectedContentHash to edit safely. For retrieval/summary
                  queries (token-budgeted projection rather than raw body),
                  use the agent-grade projection on the knowledge MCP endpoint.

get_page_history  Version history of a page, newest first. Returns
                  {exists, pageName, versions: [{version, author, lastModified,
                  changeNote}]}. Use to find version numbers for diff_page.

diff_page         Diff between two versions of a page. Returns
                  {pageName, version1, version2, diff} where diff is plain text.
                  Required: pageName, version1, version2.

get_backlinks     Pages that link to a given page.
                  Returns {pageName, backlinks: [...]} sorted alphabetically.

get_outbound_links  Pages that a given page links to.
                  Returns {pageName, links: [...]} sorted alphabetically.

get_broken_links  All references in the wiki that point to non-existent pages.
                  Returns {brokenLinks: [{pageName, referencedBy: [...]}], count}.
                  No arguments.

get_orphaned_pages  Pages with no incoming links (system pages excluded).
                  Returns {orphanedPages: [...], count}. No arguments.

get_wiki_stats    Quick dashboard: {totalPages, brokenLinkCount,
                  orphanedPageCount, recentChangesCount}. Starting point
                  before drilling into get_broken_links / get_orphaned_pages.

Write tools

write_pages       Batch-create new wiki pages. Each item: {pageName, content,
                  metadata?}. Pages that already exist are failed individually
                  (use update_page to edit those). Returns
                  {results: [{pageName, created, error?}], summary}.

update_page       Edit an existing page with optimistic locking. Required:
                  pageName, content, expectedContentHash (the contentHash from
                  the most recent read_page on this page). Optional metadata
                  is merged into frontmatter. Returns {pageName, updated,
                  newContentHash, newVersion} on success or {pageName,
                  updated:false, error:"hash mismatch", currentHash} if another
                  writer changed the page first — re-fetch and retry.

delete_pages      Batch-delete wiki pages. Required: pageNames, confirm=true.
                  System pages are refused. Pages with inbound backlinks are
                  skipped unless allowWithBacklinks=true. Returns
                  {results: [{pageName, deleted, error?, backlinks?}], summary}.

rename_page       Rename a page, moving content, attachments, and version
                  history. Requires confirm=true. updateLinks=true (default)
                  rewrites referrers to the new name. System pages cannot be
                  renamed. Returns {success, oldName, newName, finalName,
                  linksUpdated}.

mark_page_verified  Stamp pages as verified. Writes verified_at = NOW() and
                  verified_by = <verifier> into frontmatter. The structural
                  index recomputes confidence on the next save; verifiers in
                  the trusted-authors registry get confidence: authoritative.
                  Optional: pin confidence (authoritative | provisional |
                  stale). Returns {results: [{pageName, ok, verifiedAt?,
                  verifiedBy?, confidence?, error?}], succeeded, total}.

Verification & SEO tools

verify_pages      Compound check across multiple pages in one call. Default
                  checks: existence, broken_links, backlinks, outbound_links,
                  metadata_completeness. Pass checks=["seo_readiness"] (opt-in)
                  to also run summary length, tag presence, hub/related, and
                  date checks. Returns {pages: [{pageName, exists, version?,
                  outboundLinks?, brokenLinks?, backlinks?, metadata?,
                  missingMetadata?, seoWarnings?}], summary}. Use after
                  creating or updating pages to confirm integrity.

preview_structured_data  Show what frontmatter produces in the HTML layer:
                  meta description, Open Graph, JSON-LD, breadcrumb, Atom
                  feed, and Google News Sitemap eligibility. Read-only, no
                  side effects. Returns {pageName, exists, metaDescription,
                  metaKeywords?, openGraph, jsonLd, breadcrumbList?, atomFeed,
                  newsSitemap, warnings}.

ping_search_engines  Notify search engines about content changes. Pass
                  service="google_ping", "indexnow", or "all". For IndexNow,
                  pass urls=[...] for specific pages or omit to submit the
                  sitemap. Requires absolute wikantik.baseURL; IndexNow also
                  needs wikantik.indexnow.apiKey. Returns
                  {results: [{service, success, status?, urlsSubmitted?,
                  sitemapUrl?, error?}]}.

Knowledge graph proposals (when the knowledge service is wired)

list_proposals    Query knowledge proposals to check for duplicates or review
                  history. Optional filters: status (pending|approved|rejected),
                  source_page, limit, offset. Returns {proposals: [...]}.

propose_knowledge  Submit a knowledge graph proposal for human review.
                  Required: proposal_type (new-node | new-edge | new-property |
                  modify-property), proposed_data, source_page, confidence,
                  reasoning. If the relationship was previously rejected the
                  submission is declined with the rejection reason in the
                  error. Returns {id, proposal_type, source_page, status,
                  confidence, created}.

inspect_proposals  Bulk deep-dive (1..50 ids) of pending or historical KG
                  proposals. Returns full proposal, conflict flags
                  (node_exists / edge_previously_rejected), prior reviews,
                  and any linked entity snapshot. Use after list_proposals to
                  triage duplicates before approving. Required: ids[].

review_proposals  Bulk approve / reject / judge of 1..50 KG proposals.
                  Required: verdict (approve | reject | judge), ids[].
                  verdict=reject requires a top-level reason. Per-id
                  failures surface in failed[]. Approvals of proposals
                  whose source page is on the exclusion list still succeed
                  but echo a warnings_by_proposal entry.

curate_edges      Heterogeneous bulk edge ops (1..50). Required: operations[]
                  where each op has action (upsert | confirm | delete |
                  delete-and-reject), tag, and an edge id (or source/target/
                  relationship for upsert). upsert stamps HUMAN_CURATED;
                  confirm elevates an existing edge to human-curated;
                  delete-and-reject deletes and writes a rejection record so
                  the edge cannot be re-proposed.

curate_nodes      Heterogeneous bulk node ops (1..50). Required: operations[]
                  where each op has action (upsert | delete | merge), tag,
                  and a node id (or name/node_type for upsert). upsert stamps
                  HUMAN_AUTHORED; merge collapses source into target (pre-merge
                  frontmatter rewrite is handled by the REST surface only).

query_nodes       Admin-context filter+list of KG nodes. Bypasses the inclusion
                  filter so freshly-created entities (whose source pages have not
                  yet been admitted by the cluster inclusion policy) are visible.
                  Mirrors /knowledge-mcp query_nodes but with admin-bypass on.
                  Use after a curate_nodes upsert to verify the write.

search_knowledge  Admin-context fuzzy search across KG node names and property
                  text. Admin-bypass on. Use after a curate_nodes or
                  review_proposals approval to find the new entity by name.

list_orphaned_kg_nodes
                  Admin-only listing of KG nodes with no incident edges
                  (degree-0). Admin-bypass on. Filters: node_type,
                  source_page, source_page_excluded (three-state — true /
                  false / omitted distinguishes policy-hidden from truly
                  disconnected). Replaces the sample-and-walk workaround
                  for finding orphans at scale.

list_retrieval_queries
                  Read-only. Lists real retrieval queries from the query
                  log (retrieval_query_log), deduped and ranked by
                  frequency. Use max_avg_results to surface under-served
                  queries (low/zero result counts); judge true misses by
                  running candidates through the assemble_bundle MCP tool
                  — this tool makes no relevance judgment itself.
                  Parameters (all optional):
                    since_days      look back this many days (default 30)
                    actor           human | agent | unknown (default: all)
                    surface         api_bundle | api_search |
                                    mcp_assemble_bundle | tools_search_wiki
                                    (default: all)
                    max_avg_results only queries whose avg result count is
                                    <= this (find under-served queries)
                    min_occurrences minimum times seen (default 1)
                    limit           max distinct queries to return (default 50)
                  Output: { queries: [{query, occurrences, avgResultCount,
                  zeroResultCount, lastSeen}], count }.

== Concurrency model ==

Edits use optimistic concurrency, not explicit locks. The flow is:

  1. Call read_page to fetch the current content and contentHash.
  2. Compose the new content locally.
  3. Call update_page with expectedContentHash set to the hash from step 1.
  4. If another writer changed the page first, update_page returns
     {updated:false, error:"hash mismatch", currentHash}. Re-fetch with
     read_page, re-base your edits onto the new content, and retry.

This avoids long-lived locks, lets human and agent writers coexist, and makes
hash mismatches a recoverable signal rather than a hard failure.

== Completions ==

The server provides autocompletion for page-name arguments in prompts. When
using rename-page or audit-links, type the beginning of a page name and the
server suggests matching pages.

== Tips ==

- Always include a meaningful changeNote on writes (write_pages, update_page,
  delete_pages) so version history stays useful.
- For edits, fetch with read_page first, edit, then update_page with
  expectedContentHash. The hash is your concurrency guard.
- Use write_pages to create many pages in one call. Per-page results let you
  retry only the failures.
- After creating or updating pages, call verify_pages to confirm integrity in
  a single call instead of multiple individual checks.
- Run verify_pages with checks=["seo_readiness"] to audit SEO quality
  (summary length, tag presence, hub/related consistency, cluster/type).
- Use preview_structured_data after metadata edits to see exactly what meta
  tags, JSON-LD, and feed entries the page will produce.
- Stamp pages you trust with mark_page_verified — agent-grade content
  consumers treat confidence: stale as untrusted.
- Keep summaries between 50-160 characters for optimal meta descriptions.
  Pages without tags never appear in the Google News Sitemap.
